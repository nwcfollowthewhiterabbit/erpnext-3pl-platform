# Warehouse Configuration Plan

We will move in stages.

## 1. Warehouse Flows

First we will formalize the core warehouse flows that can be implemented immediately on top of standard ERPNext logic and that match the described processes:

- Receiving
- Receiving Notice
- Receiving discrepancy review
- Container / box tracking
- Putaway
- Picking / Pick List
- Packing
- Dispatch

Receiving follows this operating path:

`Receiving Notice` -> `Receiving Area` -> `Verification` -> `Putaway` -> `Storage Location`

Products are not placed directly into storage during unloading.

## 2. Product Ownership

The current business identity for a product is:

`Owner Client + Client SKU`

The visible ERPNext item code may use a readable prefix such as `CLIENTABBR-SKU`, but the system should not depend only on parsing a text prefix. The separate fields keep the current one-client-per-product model simple and leave room for a future case where two clients store the same SKU.

## 3. Warehouse-Only Interface

We will prepare the system structure for warehouse-only usage:

- hide unrelated ERP modules and entities from the interface;
- keep only warehouse operations and related flows visible;
- preserve compatibility with the standard ERPNext architecture so the system can be extended later without a large departure from default logic.

## 4. Test Flow and Demo Structure

We will prepare a test flow and demo structure:

- warehouse locations;
- barcode flow;
- client-owned products;
- demo box/container with products inside;
- demo receiving discrepancy;
- example receiving and picking process;
- demo SKU records and warehouse movements.

## 5. Warehouse Locations And Boxes

The agreed model is:

`Location = where`

`Container / Box = in what`

`Item / SKU = what`

`Quantity = how much`

ERPNext `Warehouse` records should represent only stable physical locations: zones, aisles, racks, shelves, and bins. Cardboard boxes should not be created as warehouse locations because boxes can be replaced, merged, repacked, emptied, or closed. Once an ERPNext warehouse location is used in stock transactions, deleting it is intentionally restricted.

Boxes are represented as `Three PL Container` records. A container has a client, current location, status, optional barcode / label, and item rows. This supports the real operation where staff may scan or identify a box first, then inspect products inside it.

The client can start preparing the physical warehouse location scheme now. Before entering it into ERPNext, we should agree:

- naming convention, for example `ZONE-AISLE-RACK-SHELF-BIN`;
- hierarchy depth, for example zone -> aisle -> rack -> shelf -> bin;
- whether mixed clients or mixed SKUs are allowed in one bin;
- label format for locations and containers.

## 6. Test Access and Manual

After that we will send access to the test system with a short description:

- what processes are already configured;
- how the current flow works;
- where to go inside the system;
- where the corresponding sections and manual flow are located.

## Open Questions

The next details to clarify are:

- Will separate scanner devices be used, or is browser plus barcode scanner enough for the first stage?
- Is multi-client inventory ownership required?
- Are serial numbers, batch numbers, or expiry dates required?
- Should storage be fixed-location or dynamic putaway?
- Will courier or shipping system integrations be needed later?
- Should the primary warehouse workflow be mobile/scanner-first?
- Should container labels be generated by ERPNext or imported from an external warehouse label process?
- What exact warehouse location naming convention should be used?
- Should repacking be allowed from the first production release, or can it be manual at MVP stage?
