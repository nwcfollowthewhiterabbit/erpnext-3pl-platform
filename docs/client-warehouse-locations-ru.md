# Схема складских локаций и коробок

## Что уже согласовали

В ERPNext складские локации и коробки нужно вести отдельно.

Базовая модель:

- Локация = где лежит товар.
- Контейнер / коробка = в чем лежит товар.
- Товар / SKU = что лежит.
- Количество = сколько лежит.

Например:

- `A-01-R01-S02-B03` - физическая складская ячейка.
- `BOX-ALPHA-001` - коробка, которая сейчас находится в этой ячейке.
- `ALPHA-SKU-001` - товар внутри коробки.
- `10 pcs` - количество товара внутри коробки.

## Почему коробки не должны быть складскими локациями

Коробка - временная сущность. Ее можно заменить, объединить с другой коробкой, переупаковать, опустошить или закрыть.

Складская локация - стабильное физическое место: зона, ряд, стеллаж, полка, ячейка. После складских операций ERPNext не дает просто удалить использованную локацию, поэтому если создавать каждую коробку как складскую локацию, дерево склада быстро станет неудобным и технически проблемным.

Правильнее:

- зоны, ряды, стеллажи, полки и ячейки вести как `Warehouse`;
- коробки вести как `Three PL Container`;
- у коробки хранить текущую локацию, клиента, статус, штрихкод/лейбл и товары внутри.

В терминах WMS `Three PL Container` - это наш слой Handling Units. В стандартном ERPNext есть базовая складская логика, но нет полноценного готового 3PL Handling Unit процесса для коробок, палет, переупаковки и истории перемещений. Поэтому этот функционал создается как кастомный слой поверх стандартного ERPNext.

## Что уже реализовано

Уже есть базовая модель:

- складская иерархия для receiving, inspection, storage, packing и shipping;
- `Three PL Container` для коробок/контейнеров;
- привязка контейнера к клиенту;
- тип контейнера: box, carton, pallet, tote, other;
- текущая складская локация контейнера;
- статусы жизненного цикла: expected, received, in verification, ready for putaway, stored, picking, picked, packed, shipped, empty, closed, replaced;
- связь с parent container и replaced by для палетизации и переупаковки;
- товары внутри контейнера;
- ссылки на контейнеры в приемке, putaway, picking, packing и отчетах;
- demo-контейнеры и demo-данные для Alpha Client.

Это уже можно использовать для обсуждения и проверки концепции.

## Что еще нужно доделать

Осталось превратить базовую модель в полноценный операционный процесс:

- согласовать naming convention для локаций;
- создать или импортировать реальное дерево склада;
- добавить историю перемещений контейнеров;
- добавить сценарий repack, например две маленькие коробки заменили одной большой;
- добавить статусы и действия `empty`, `closed`, `replaced`;
- сделать scanner/mobile UX: сначала скан локации, потом коробки, потом товара и количества;
- автоматизировать обновление inventory snapshot после складских движений и движений контейнеров.

## Что можно делать клиенту сейчас

Клиент может начинать готовить схему локаций склада, но перед внесением в ERPNext нужно согласовать:

- глубину иерархии: зона -> ряд -> стеллаж -> полка -> ячейка;
- формат названий, например `ZONE-AISLE-RACK-SHELF-BIN`;
- будут ли смешиваться разные клиенты в одной ячейке;
- будут ли смешиваться разные SKU в одной ячейке;
- как будут печататься и сканироваться лейблы локаций и коробок.

Рекомендация: не создавать коробки как локации. Коробки должны быть контейнерами, а реальные физические места на складе - локациями.
