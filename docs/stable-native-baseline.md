# Stable Native Baseline

Status date: 2026-06-17.

This is the clean stable starting point for MVP1 manual testing.

The project is now a native ERPNext Desk solution. The previous custom client interface, forced redirects, and route-specific nginx behavior are out of the active MVP path. nginx should remain a neutral HTTPS reverse proxy to the ERPNext frontend.

## Access Model

Access is controlled in ERPNext/Frappe, not in nginx:

- role `home_page`;
- user `default_workspace`;
- Module Profile;
- Workspace roles;
- DocType permissions;
- Customer `User Permission`;
- narrow data guards for customer-owned client documents and client-visible system data.

Canonical entry points:

- Client: `/desk/3pl-client`;
- Warehouse operator/manager: `/desk/3pl-warehouse`;
- Business owner: `/desk/3pl-warehouse`.

## Current Stable Constraints

- `3PL Client` is a Desk `System User`, not a legacy Website Portal user.
- `3PL Client` uses Module Profile `3PL Client Only`.
- Warehouse users use Module Profile `Warehouse Only`.
- Client and warehouse workspaces are role-scoped.
- The client user must not have `Customer`, `Stock User`, `Stock Manager`, `System Manager`, or warehouse roles.
- Client data access is scoped to `Demo Client Alpha` through Customer User Permission and app-level document guards.
- Client system access is intentionally narrow: the client may not list unrelated `User` records and may not read `Role`, `DocType`, `Module Profile`, or `System Settings`.

## Native Report And Sidebar Rules

ERPNext reports must be exposed through standard Desk reports, permissions, Workspace links, and Workspace Sidebar configuration. Do not add custom report pages or nginx routes for MVP report access.

Required report pattern:

- The `Report` must have the correct `ref_doctype`.
- Every role that should open the report must have `read = 1` and `report = 1` on that `ref_doctype`.
- If the report reads related source DocTypes directly, those source DocTypes must also have the required role permissions or the query must intentionally apply server-side filtering.
- Workspace report links must include the report target and reference DocType metadata. In Workspace JSON, report links should set `link_type = "Report"`, `link_to = "<Report Name>"`, and `report_ref_doctype = "<Report Ref DocType>"`.
- Client-facing reports must filter by the current user's allowed customer/client. Never rely only on hidden UI links for data isolation.

Current MVP report reference DocTypes:

| Report | Ref DocType |
| --- | --- |
| `3PL Client Inventory Summary` | `Three PL Inventory Snapshot` |
| `3PL Inventory Balance By Date` | `Three PL Inventory Balance Snapshot` |
| `3PL Warehouse Operation Turnover` | `Three PL Container Movement` |
| `3PL Receiving Discrepancies` | `Inbound Shipment Notice` |

Workspace Sidebar rule:

The module `ERPNext 3PL` is shared by client and warehouse roles. Do not rely on ERPNext's autogenerated module sidebar for this module. Autogenerated sidebars can include `Workspace` links that one role cannot access. In ERPNext v16 this can crash Desk report pages before the datatable renders, with a browser error like:

```text
TypeError: Cannot read properties of undefined (reading 'public')
```

Keep an explicit `Workspace Sidebar` named `ERPNext 3PL` with role-scoped `display_depends_on` values for role-specific Workspace links. The sidebar is a site configuration record (`standard = 0`) and must be kept in the app fixtures for repeatability.

After changing `Workspace Sidebar` records, `bench clear-cache` may not be enough. ERPNext's autogenerated sidebar uses in-process `site_cache`. Restart the live backend service after sidebar changes, then test with a fresh browser session.

Minimum verification after adding or changing a report:

- Log in as `alpha.client@example.test`.
- Open the report route under `/desk/query-report/<Report Name>`.
- Confirm the page has a datatable, not just a successful API response.
- Confirm browser console has no `Cannot read properties of undefined (reading 'public')` page error.
- Repeat at least one report check as `warehouse.demo@example.test` and `warehouse.manager@example.test` when the report is warehouse-visible.

## Known Naming Debt

Some internal DocType fields still use legacy names such as `portal_source` and `portal_items_description`. They are compatibility/storage fields for client-originated Desk records, not an active custom portal. Renaming them should be handled later as a controlled schema migration, outside the clean baseline stabilization.

## Explicit Backlog

No server-side route/landing redirect guard is part of this baseline.

If manual testing shows that users often land in the wrong workspace after login or bookmarks, add a small Frappe-native route consistency guard later. That guard should only normalize landing routes; it must not reintroduce a custom client interface or nginx-based business routing.
